# 일반적인 리스크 패턴 및 대응 방안

AI Agent 프로젝트에서 자주 발생하는 리스크 패턴과 대응 전략입니다.

---

## 1. 데이터 관련 리스크

### 패턴 1: "API가 있다고 했는데..."

**증상**: API 존재하나 실제로 필요한 데이터 없음

**징후**:
- API 문서가 오래됨
- "내부용 API라서..."
- 필요한 필드가 응답에 없음

**대응**:
1. 실제 API 테스트 필수 (문서만 믿지 말 것)
2. 샘플 데이터로 파이프라인 검증
3. 대체 데이터 소스 Plan B 준비

### 패턴 2: "데이터는 있는데 품질이..."

**증상**: 데이터 존재하나 결측/오류 많음

**징후**:
- "수동 입력이라..."
- 필드 값 불일치
- 인코딩/포맷 문제

**대응**:
1. 데이터 품질 체크 스크립트 준비
2. 정제 파이프라인 구축 일정 추가
3. 품질 임계치 정의 (80% 이상이면 진행 등)

---

## 2. 판단 기준 관련 리스크

### 패턴 3: "전문가마다 다르게 판단"

**증상**: 동일 케이스에 다른 결정

**징후**:
- "그때그때 달라요"
- "A팀장님은 이렇게 하는데..."
- 문서화된 기준 없음

**대응**:
1. 판단 기준 합의 회의 진행
2. 경계 케이스 목록 작성
3. Golden Dataset 구축 (정답이 합의된 예시)

### 패턴 4: "예외가 너무 많아요"

**증상**: 규칙은 있으나 예외가 규칙보다 많음

**징후**:
- "보통은 이런데 이 경우는..."
- 예외 처리 로직이 본 로직보다 큼
- "VIP는 다르게 처리"

**대응**:
1. 예외를 규칙화 (예외도 패턴 있음)
2. 예외 빈도 분석 (상위 10개만 처리)
3. 예외 발생 시 Human 에스컬레이션

---

## 3. 정확도 관련 리스크

### 패턴 5: "90%면 되죠?"

**증상**: 10% 오류의 실제 영향 과소평가

**징후**:
- 오류 케이스 분석 안 함
- "나중에 고치면 되지"
- 롤백 계획 없음

**대응**:
1. 오류 영향 시뮬레이션
2. 오류 유형별 대응 정의
3. 모니터링 + 알림 시스템 구축

### 패턴 6: "사람이 검토하면 되니까"

**증상**: 검토 병목 미예상

**징후**:
- 검토자 리소스 미확인
- "하루에 100건 정도면..."
- 실제로는 1000건 발생

**대응**:
1. 예상 볼륨 × 검토 시간 계산
2. 샘플링 검토 방식 설계
3. 자동 필터링으로 검토 대상 축소

## 4. 성능 관련 리스크

### 패턴 7: "프로토타입은 빨랐는데..."

**증상**: 실제 환경에서 성능 저하

**징후**:
- 테스트 데이터 10건
- 실제 데이터 10만건
- "병렬 처리하면 되겠죠"

**대응**:
1. 실제 규모 데이터로 테스트
2. 병목 구간 사전 식별
3. 스케일링 계획 수립

### 패턴 8: "LLM이 느려서..."

**증상**: LLM 응답 시간이 SLA 미달

**징후**:
- 복잡한 프롬프트
- 긴 컨텍스트
- 동기 처리 의존

**대응**:
1. 프롬프트 최적화 (토큰 수 감소)
2. 캐싱 전략 (유사 요청 재사용)
3. 비동기 + 알림 패턴 도입

---

## 5. 통합 관련 리스크

### 패턴 9: "권한이 필요한데..."

**증상**: API 호출 권한 미확보

**징후**:
- "정보보안팀 승인 필요"
- "프로덕션 접근은..."
- OAuth/SSO 연동 미고려

**대응**:
1. 권한 획득 프로세스 사전 확인
2. 최소 권한 원칙으로 설계
3. 개발/스테이징 환경 먼저 확보

### 패턴 10: "시스템이 변경됐어요"

**증상**: 연동 시스템 변경으로 파이프라인 파손

**징후**:
- API 버전 관리 안 됨
- 알림 없는 변경
- 스키마 변경 빈번

**대응**:
1. API 버전 고정 (v1, v2 명시)
2. 스키마 검증 로직 추가
3. 통합 테스트 자동화
