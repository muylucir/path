# Coding Agent (코딩 에이전트)

## 개념

Coding Agent는 코드에 대해 추론하고, 코드를 생성/수정하며, 개발 환경과 상호작용하는 Agent 패턴입니다. 단순한 코드 생성을 넘어, 현재 코드베이스의 맥락을 이해하고, 변경 사항을 계획하며, 실행/테스트/검증의 피드백 루프를 통해 반복적으로 코드를 개선합니다.

```
Query → Extract Environment Context → LLM Reasoning → Plan Changes → Execute Actions → Verify → Response
```

## 아키텍처 흐름

```
[사용자 요청]
     │
     ▼
[환경 컨텍스트 수집]
  ├── 활성 파일, 커서 위치, 선택 영역
  ├── 프로젝트 구조, 의존성 정보
  ├── 에러 메시지, 린트 결과
  ├── Git 상태 (변경 파일, 브랜치)
  └── 관련 파일 / 심볼 참조
     │
     ▼
[LLM 추론]
  ├── 요청 의도 파악
  ├── 영향 범위 분석
  ├── 변경 계획 수립
  └── 코드 변경 사항 생성
     │
     ▼
[액션 실행]
  ├── 코드 삽입 / 수정 / 삭제
  ├── 파일 생성 / 이동
  ├── 테스트 실행
  ├── 린팅 / 포매팅
  └── 커맨드 실행
     │
     ▼
[검증 및 피드백]
  ├── 테스트 결과 확인
  ├── 에러 발생 시 수정 루프
  └── 사용자 확인 요청
     │
     ▼
[최종 결과]
```

## 핵심 구성요소

### 1. Environment Context Extraction (환경 컨텍스트 수집)

| 컨텍스트 유형 | 설명 | 활용 |
|-------------|------|------|
| **활성 파일** | 현재 열려 있거나 편집 중인 파일 | 변경 대상 식별 |
| **커서/선택 영역** | IDE에서의 현재 위치 | 정확한 수정 위치 결정 |
| **구문 트리 (AST)** | 코드의 구조적 분석 | 함수/클래스 경계 이해 |
| **에러/경고** | 컴파일러, 린터, 테스트 오류 | 문제 원인 파악 |
| **프로젝트 구조** | 디렉토리, 설정 파일, 의존성 | 코드베이스 이해 |
| **Git 상태** | 변경 파일, 커밋 히스토리 | 변경 범위 관리 |
| **심볼 참조** | 함수/변수의 정의와 사용처 | 리팩토링 영향 분석 |

### 2. LLM Reasoning (추론 엔진)

```
[컨텍스트 입력]
     │
     ▼
[의도 파악] → "무엇을 변경/생성/수정해야 하는가?"
     │
     ▼
[영향 분석] → "어떤 파일과 함수가 영향을 받는가?"
     │
     ▼
[변경 계획] → "어떤 순서로, 어떻게 변경하는가?"
     │
     ▼
[코드 생성] → 실제 코드 diff 또는 새 코드 생성
```

### 3. Action Execution (액션 실행)

| 액션 유형 | 설명 | 예시 |
|----------|------|------|
| **코드 삽입** | 새 코드 추가 | 새 함수, 클래스, 모듈 생성 |
| **코드 수정** | 기존 코드 변경 | 버그 수정, 리팩토링 |
| **코드 삭제** | 불필요한 코드 제거 | 데드 코드, 중복 코드 |
| **파일 관리** | 파일 생성/이동/삭제 | 모듈 분리, 구조 변경 |
| **테스트 실행** | 테스트 수행 및 결과 확인 | 유닛 테스트, 통합 테스트 |
| **빌드/린트** | 빌드 및 정적 분석 | 컴파일 에러, 스타일 검사 |
| **커맨드 실행** | 셸 명령 실행 | 패키지 설치, 마이그레이션 |

### 4. Feedback Loop (피드백 루프)

```
[코드 변경 적용]
     │
     ▼
[자동 검증]
  ├── 구문 오류 체크
  ├── 린트 검사
  ├── 테스트 실행
  └── 타입 체크
     │
     ├── 통과 → [완료]
     │
     └── 실패 → [에러 분석]
                  │
                  ▼
               [수정 계획] → [코드 재수정] → [자동 검증] (반복)
```

## Description (단계별)

### Step 1: 요청 수신 및 컨텍스트 수집

```
User: "UserService 클래스에 이메일 유효성 검증 메서드를 추가해줘"

[컨텍스트 수집]
  ├── 파일: src/services/user_service.py (현재 열린 파일)
  ├── 클래스: UserService (Line 15-120)
  ├── 기존 메서드: create_user(), get_user(), update_user()
  ├── 의존성: email-validator 패키지 (requirements.txt에 없음)
  ├── 테스트: tests/test_user_service.py 존재
  └── 스타일: PEP 8, type hints 사용 중
```

### Step 2: 변경 계획 수립

```
[Plan]
  1. requirements.txt에 email-validator 의존성 추가
  2. UserService 클래스에 validate_email() 메서드 추가
     - 기존 코드 스타일에 맞춰 type hints 포함
     - docstring 작성
  3. tests/test_user_service.py에 테스트 케이스 추가
     - 유효한 이메일 테스트
     - 무효한 이메일 테스트
     - 빈 문자열 / None 테스트
  4. 기존 create_user() 메서드에서 validate_email() 호출 추가
```

### Step 3: 코드 변경 실행

```
[Action 1] 파일 수정: requirements.txt
  + email-validator>=2.0.0

[Action 2] 파일 수정: src/services/user_service.py
  + def validate_email(self, email: str) -> bool:
  +     """이메일 주소의 유효성을 검증합니다."""
  +     ...

[Action 3] 파일 수정: tests/test_user_service.py
  + def test_validate_email_valid(): ...
  + def test_validate_email_invalid(): ...
  + def test_validate_email_empty(): ...

[Action 4] 파일 수정: src/services/user_service.py
  기존 create_user() 메서드에 validate_email() 호출 추가
```

### Step 4: 검증 및 피드백 루프

```
[검증 1] 린트 실행 → 통과
[검증 2] 타입 체크 → 통과
[검증 3] 테스트 실행 → 1건 실패
     │
     ▼
[에러 분석] test_validate_email_empty: None 입력 시 AttributeError
     │
     ▼
[수정] validate_email()에 None 체크 추가
     │
     ▼
[검증 재실행] 테스트 전체 통과
     │
     ▼
[완료] 변경 사항 요약 리포트 제공
```

## 역량 (Capabilities)

| 역량 | 설명 |
|------|------|
| **높은 맥락 인식** | 코드베이스 전체 구조를 이해하고 적절한 위치에 코드 배치 |
| **반복적 추론** | 에러 발생 시 원인을 분석하고 수정을 반복 |
| **계획과 실행 분리** | 무엇을 변경할지 계획한 후 실행하여 일관성 보장 |
| **코드 스타일 적응** | 기존 코드베이스의 스타일, 패턴, 컨벤션을 학습하여 적용 |
| **멀티 파일 수정** | 여러 파일에 걸친 일관된 변경 수행 |
| **자가 검증** | 생성/수정한 코드를 자동으로 테스트하여 품질 보장 |

## 고급 패턴

### Plan-and-Execute 분리

```
[Planning Phase]
  Agent에게 "코드를 수정하지 말고, 변경 계획만 수립하라" 지시
     │
     ▼
  [변경 계획]
  1. 파일 A, Line 42: 함수 시그니처 변경
  2. 파일 B, Line 15-30: 새 메서드 추가
  3. 파일 C: 테스트 추가
     │
     ▼
[Review Phase]
  사용자가 계획 검토 및 승인
     │
     ▼
[Execution Phase]
  승인된 계획에 따라 코드 변경 실행
```

### Multi-file Refactoring

```
[리팩토링 요청] → "모놀리식 모듈을 분리해줘"
     │
     ▼
[의존성 그래프 분석]
  ├── 모듈 간 의존성 매핑
  ├── 순환 참조 감지
  └── 최적 분리 지점 결정
     │
     ▼
[분리 계획]
  ├── 새 모듈 구조 설계
  ├── import 경로 업데이트
  ├── 인터페이스 정의
  └── 테스트 재배치
     │
     ▼
[순차 실행] → 파일 생성 → 코드 이동 → import 수정 → 테스트 검증
```

### Self-healing Loop

```
[코드 생성/수정]
     │
     ▼
[자동 검증] ← 린트, 타입체크, 테스트
     │
     ├── 통과 → [완료]
     │
     └── 실패 → [에러 수집]
                  │
                  ▼
               [에러 컨텍스트 + 실패한 코드를 LLM에 전달]
                  │
                  ▼
               [수정된 코드 생성] → [자동 검증] (최대 N회 반복)
```

## 주의사항

1. **보안 (코드 인젝션)**: Agent가 생성한 코드가 악의적 명령을 포함할 수 있음 — 샌드박스 환경에서 실행, 위험 패턴 필터링
2. **존재하지 않는 API 생성 (환각)**: LLM이 존재하지 않는 라이브러리 함수를 사용할 수 있음 — 생성 후 import 검증, 의존성 체크
3. **변경 전 테스트**: 코드 수정을 적용하기 전에 반드시 테스트로 검증 — 되돌릴 수 없는 변경 방지
4. **컨텍스트 윈도우 한계**: 대규모 코드베이스 전체를 컨텍스트에 넣을 수 없음 — 관련 부분만 선별적으로 수집
5. **권한 제어**: Agent가 실행할 수 있는 명령과 수정할 수 있는 파일 범위를 제한 — 화이트리스트 기반 접근
6. **비결정적 출력**: 동일 요청에 다른 코드를 생성할 수 있음 — 테스트 기반 검증으로 품질 보장
7. **대규모 변경의 위험성**: 많은 파일을 한 번에 수정하면 디버깅이 어려움 — 단계적 변경 및 중간 검증
8. **기존 코드 스타일 무시**: Agent가 코드베이스의 기존 패턴을 따르지 않을 수 있음 — 스타일 가이드를 컨텍스트에 포함
