# Memory Agent (메모리 증강 에이전트)

## 개념

Memory Agent는 단기 기억(STM)과 장기 기억(LTM)을 활용하여 세션 간 맥락을 유지하고, 과거 경험으로부터 학습하며, 개인화된 상호작용을 제공하는 Agent 패턴입니다. LLM의 컨텍스트 윈도우 한계를 극복하고, 지속적인 상태 유지를 통해 시간이 지남에 따라 더 나은 서비스를 제공합니다.

```
Receive Input → Retrieve STM → Retrieve LTM → Reason with Memory Context → Generate Output → Update Memory
```

## 아키텍처 흐름

```
[사용자 입력]
     │
     ▼
[메모리 검색 단계]
  ├── STM 조회: 최근 대화, 세션 컨텍스트 (키-값 저장소 / 인메모리 캐시)
  ├── LTM-Structured 조회: 사실, 관계, 이력 (관계형/문서 DB)
  ├── LTM-Semantic 조회: 의미 기반 유사 경험 (벡터 데이터베이스)
  └── Working Memory 조회: 현재 태스크 진행 상태
     │
     ▼
[컨텍스트 구성]
  ├── 메모리에서 검색된 관련 정보 조합
  ├── 최근 히스토리 요약
  └── 현재 태스크 상태 반영
     │
     ▼
[LLM 추론] ← 메모리 증강 컨텍스트 기반
     │
     ▼
[응답 생성]
     │
     ▼
[메모리 업데이트]
  ├── STM 업데이트: 현재 대화 추가
  ├── LTM 업데이트: 새로운 사실/관계 저장
  └── Working Memory 업데이트: 태스크 진행 상태 갱신
```

## 핵심 구성요소

### 1. 메모리 유형

#### Short-term Memory (STM) - 단기 기억

| 항목 | 설명 |
|------|------|
| **저장 내용** | 최근 대화 내역, 현재 세션의 컨텍스트, 임시 상태 |
| **저장소** | 인메모리 캐시, 키-값 저장소 |
| **수명** | 세션 단위 (세션 종료 시 만료 또는 TTL 기반) |
| **접근 속도** | 매우 빠름 (밀리초 단위) |
| **용도** | 대화 연속성 유지, 즉각적 맥락 참조 |

```
STM 구조 예시:
{
  "session_id": "sess-abc123",
  "user_id": "user-456",
  "messages": [
    { "role": "user", "content": "...", "timestamp": "..." },
    { "role": "assistant", "content": "...", "timestamp": "..." }
  ],
  "session_context": {
    "current_topic": "프로젝트 일정 논의",
    "mentioned_entities": ["Sprint 5", "배포 일정"],
    "user_mood": "neutral"
  },
  "ttl": 3600
}
```

#### Long-term Memory (LTM) - Structured (구조적 장기 기억)

| 항목 | 설명 |
|------|------|
| **저장 내용** | 사실(facts), 관계(relationships), 사용자 프로필, 이력 로그 |
| **저장소** | 관계형 데이터베이스, 문서 데이터베이스 |
| **수명** | 영구적 (명시적 삭제 또는 정책 기반 정리) |
| **접근 속도** | 빠름 (인덱싱된 쿼리) |
| **용도** | 사용자 선호도, 과거 결정 이력, 팀/프로젝트 정보 |

```
LTM-Structured 예시:
{
  "user_id": "user-456",
  "facts": [
    { "key": "preferred_language", "value": "Python", "confidence": 0.95, "updated": "..." },
    { "key": "team", "value": "Backend Engineering", "confidence": 1.0, "updated": "..." },
    { "key": "timezone", "value": "Asia/Seoul", "confidence": 1.0, "updated": "..." }
  ],
  "relationships": [
    { "subject": "user-456", "relation": "works_on", "object": "project-alpha" },
    { "subject": "user-456", "relation": "reports_to", "object": "user-789" }
  ],
  "interaction_log": [
    { "date": "2025-03-01", "summary": "프로젝트 알파 아키텍처 논의, 마이크로서비스 방식 합의" }
  ]
}
```

#### Long-term Memory (LTM) - Semantic (의미적 장기 기억)

| 항목 | 설명 |
|------|------|
| **저장 내용** | 임베딩 벡터로 변환된 과거 대화, 문서, 경험 |
| **저장소** | 벡터 데이터베이스 |
| **수명** | 영구적 (메모리 용량 관리 정책에 따름) |
| **접근 속도** | 보통 (벡터 유사도 검색) |
| **용도** | 유사 경험 검색, 맥락적 회상, RAG 기반 기억 |

```
LTM-Semantic 예시:
{
  "memory_id": "mem-001",
  "content": "사용자가 3월에 API 게이트웨이 설계를 논의했으며, REST보다 GraphQL을 선호한다고 언급",
  "embedding": [0.12, -0.45, 0.78, ...],
  "metadata": {
    "user_id": "user-456",
    "topic": "API 설계",
    "date": "2025-03-15",
    "importance": 0.8
  }
}
```

#### Working Memory (작업 기억)

| 항목 | 설명 |
|------|------|
| **저장 내용** | 현재 태스크 진행 상태, 중간 결과, 계획 |
| **저장소** | 인메모리 또는 키-값 저장소 |
| **수명** | 태스크 완료까지 |
| **접근 속도** | 매우 빠름 |
| **용도** | 복합 태스크의 단계 추적, 중간 산출물 관리 |

```
Working Memory 예시:
{
  "task_id": "task-789",
  "goal": "보고서 자동 생성",
  "plan": ["데이터 수집", "분석", "시각화", "문서 작성"],
  "current_step": 2,
  "intermediate_results": {
    "data_collected": { "rows": 1500, "source": "sales_db" },
    "analysis_in_progress": true
  },
  "blockers": []
}
```

### 2. Memory Injection Strategy (메모리 주입 전략)

```
[프롬프트 구성]

System Prompt:
  "당신은 사용자의 이전 대화와 선호도를 기억하는 어시스턴트입니다."

Memory Context Section:
  === 사용자 프로필 ===
  - 이름: 김개발
  - 선호 언어: Python
  - 팀: Backend Engineering

  === 관련 과거 대화 요약 ===
  - [3/15] API 설계 논의: GraphQL 선호, 인증은 OAuth2.0 방식 합의
  - [3/18] 데이터베이스 마이그레이션 진행: PostgreSQL → 완료

  === 현재 세션 요약 ===
  - Sprint 5 일정 논의 중
  - 배포 일정이 주요 관심사

  === 현재 태스크 상태 ===
  - 진행 중: 배포 체크리스트 작성 (2/5 단계 완료)

User Message:
  "이전에 논의했던 API 인증 방식 기억나? 그걸로 배포 체크리스트에 추가해줘"
```

### 3. Memory Lifecycle Management (메모리 수명 관리)

```
[메모리 생성]
     │
     ▼
[활성 사용] ← 검색, 참조, 업데이트
     │
     ▼
[중요도 평가] ← 접근 빈도, 시간 경과, 관련성
     │
     ├── 중요 → [장기 보존]
     │
     └── 비중요 → [압축/요약] → [아카이브] → [삭제]
```

## Description (단계별)

### Step 1: 입력 수신 및 메모리 검색

```
[사용자 입력] → "지난번 논의했던 마이크로서비스 아키텍처 결정사항을 정리해줘"
     │
     ▼
[STM 검색] → 현재 세션에서 관련 대화 내역 확인
     │
     ▼
[LTM-Structured 검색] → 사용자의 "아키텍처 결정" 관련 fact 조회
     │
     ▼
[LTM-Semantic 검색] → "마이크로서비스 아키텍처" 임베딩으로 유사 기억 검색
     │
     ▼
[검색 결과 통합]
  ├── STM: 현재 세션에서 관련 대화 없음
  ├── LTM-Structured: "2025-03-01 마이크로서비스 방식 합의" fact 발견
  └── LTM-Semantic: 2건의 관련 대화 기록 검색
      ├── [3/1] "모놀리식에서 마이크로서비스로 전환 논의..."
      └── [3/10] "서비스 경계 정의: 사용자, 주문, 결제 분리..."
```

### Step 2: 컨텍스트 구성 및 추론

```
[메모리 컨텍스트 구성]
  관련 기억을 시간순 정렬 후 프롬프트에 주입

[LLM 추론]
  과거 대화 기록을 바탕으로 결정사항 정리

[응답 생성]
  "지난 3월 논의하신 마이크로서비스 아키텍처 결정사항을 정리하면:
   1. 모놀리식 → 마이크로서비스 전환 결정 (3/1)
   2. 서비스 경계: 사용자, 주문, 결제 3개 서비스로 분리 (3/10)
   ..."
```

### Step 3: 메모리 업데이트

```
[새 메모리 생성]
  ├── STM: 현재 대화 내용 추가
  ├── LTM-Structured: "아키텍처 결정사항 정리 완료" fact 추가
  └── LTM-Semantic: 현재 대화 요약을 임베딩하여 저장
```

## 역량 (Capabilities)

| 역량 | 설명 |
|------|------|
| **세션 연속성** | 세션이 끊겨도 이전 대화의 맥락을 유지하여 자연스러운 대화 |
| **목표 지속성** | 장기 태스크의 목표와 진행 상태를 기억하여 중단 후 재개 가능 |
| **맥락적 인식** | 사용자의 상황, 역할, 관심사를 종합적으로 이해 |
| **적응적 학습** | 과거 결과로부터 패턴을 학습하여 점점 더 나은 응답 |
| **개인화** | 사용자별 선호도, 스타일, 이력을 반영한 맞춤형 상호작용 |
| **경험 기반 추론** | 유사한 과거 상황의 결과를 참고하여 더 나은 판단 |

## 적합 사례

| 사례 | 설명 |
|------|------|
| **대화형 코파일럿** | 사용자 선호도를 기억하고, 과거 요청 패턴에 맞춰 제안 |
| **코딩 Agent** | 코드베이스 변경 이력을 추적하고, 이전 결정사항을 참고하여 일관된 코드 생성 |
| **워크플로우 Agent** | 반복 태스크의 과거 실행 결과를 학습하여 프로세스 최적화 |
| **학습 튜터** | 학습자의 진도, 강점/약점, 학습 스타일을 기억하여 맞춤 교육 |
| **프로젝트 관리 Agent** | 팀 회의 결정사항, 태스크 이력, 팀원 전문 분야를 기억 |

```
│  │              │  │ │Structured│ │Semantic ││  │
│  │  - 세션 데이터 │  │ │(문서 DB)  │ │(벡터 DB)││  │
│  │  - 최근 대화  │  │ │ - Facts  │ │ -임베딩 ││  │

│                    └─────────────────────────┘  │
│                                                 │

│  │ Working Memory   │                            │
│  │ (인메모리/캐시)   │                            │

│  └─────────────────┘                            │
└────────────────────────────────────────────────┘
```

### 메모리 검색 전략

| 전략 | 방식 | 적합 상황 |
|------|------|----------|
| **최신성 기반** | 가장 최근 기억 우선 | 현재 대화 맥락 유지 |
| **관련성 기반** | 의미적 유사도로 검색 | 과거 유사 경험 참조 |
| **중요도 기반** | 중요도 점수가 높은 기억 우선 | 핵심 사실/결정사항 참조 |
| **하이브리드** | 최신성 + 관련성 + 중요도 결합 | 복합 시나리오 |

### 메모리 압축 및 요약

```
[긴 대화 히스토리]
     │
     ▼
[요약 생성]
  ├── 최근 N턴: 원본 유지
  ├── 이전 대화: LLM으로 요약 → 요약본 저장
  └── 오래된 대화: 핵심 fact만 추출 → LTM-Structured에 저장
     │
     ▼
[압축된 컨텍스트]
  "=== 이전 대화 요약 ===
   3/1: 마이크로서비스 전환 결정
   3/10: 서비스 경계 확정
   === 최근 대화 (원본) ===
   [최근 5턴 원본 대화]"
```

### Agent 구성 예시

```
Memory Agent
├── System Prompt
│   ├── 역할: 메모리 증강 어시스턴트
│   ├── 메모리 활용 지침
│   └── 개인정보 보호 규칙
├── Memory Tools
│   ├── recall_recent: STM에서 최근 대화 검색
│   ├── recall_facts: LTM-Structured에서 fact 조회
│   ├── recall_similar: LTM-Semantic에서 유사 기억 검색
│   ├── save_fact: 새로운 fact 저장
│   ├── save_memory: 장기 기억에 저장
│   └── get_task_state: Working Memory 조회
├── Memory Management
│   ├── Compression: 오래된 대화 요약
│   ├── Cleanup: 만료된 기억 정리
│   └── Deduplication: 중복 기억 제거
└── Privacy Controls
    ├── PII 필터링
    ├── 접근 권한 관리
    └── 삭제 요청 처리
```

## 고급 패턴

### Reflective Memory (반성적 기억)

```
[상호작용 완료]
     │
     ▼
[반성 단계] ← "이 상호작용에서 무엇을 배웠는가?"
  ├── 새로 알게 된 사실
  ├── 사용자 선호도 업데이트
  ├── 성공/실패 패턴
  └── 향후 개선점
     │
     ▼
[메타 기억 저장] → "사용자가 코드 리뷰 시 보안 관점을 항상 요청함"
```

### Episodic Memory (일화적 기억)

```
[에피소드 단위 저장]
  Episode 1: "프로젝트 킥오프" (3/1)
    ├── 참여자, 결정사항, 액션 아이템
    └── 감정/분위기: 긍정적

  Episode 2: "첫 스프린트 회고" (3/15)
    ├── 성과, 문제점, 개선사항
    └── 감정/분위기: 건설적
```

### Forgetting Strategy (망각 전략)

```
[메모리 정리 정책]
  ├── 시간 기반: 90일 미접근 기억 아카이브
  ├── 용량 기반: 사용자당 최대 메모리 크기 제한
  ├── 중요도 기반: 중요도 0.3 미만 기억 정리
  └── 명시적 삭제: 사용자 요청에 의한 삭제
```

## 주의사항

1. **메모리 크기 관리**: 메모리가 무한히 증가하면 검색 성능 저하 및 비용 증가 — 압축, 요약, 정리 정책 필수
2. **일관성 유지**: 동일 사실에 대해 여러 기억이 충돌할 수 있음 — 최신 기억 우선, 충돌 해결 로직 구현
3. **민감 데이터 보호**: 개인정보, 비밀번호, 금융 정보 등이 메모리에 저장될 위험 — PII 필터링, 암호화, 접근 제어
4. **정리 정책 (Cleanup)**: 오래되거나 불필요한 기억의 자동 정리 — TTL 설정, 주기적 가비지 컬렉션
5. **메모리 환각**: Agent가 저장하지 않은 기억을 "만들어낼" 위험 — 메모리 검색 결과를 명시적으로 인용하도록 유도
6. **컨텍스트 윈도우 초과**: 많은 기억을 프롬프트에 주입하면 윈도우 초과 — 관련성 기반 선별, 요약 압축
7. **다중 사용자 격리**: 사용자 간 메모리가 섞이지 않도록 엄격한 격리 — 사용자 ID 기반 파티셔닝
8. **메모리 업데이트 빈도**: 매 턴마다 LTM을 업데이트하면 비용과 지연 증가 — 배치 업데이트 또는 중요도 기반 선별 업데이트
