# RAG Agent (Retrieval-Augmented Generation)

## 개념

RAG Agent는 정보 검색(Retrieval)과 텍스트 생성(Generation)을 결합하여, 외부 지식 소스를 기반으로 정확하고 맥락에 맞는 응답을 생성하는 패턴입니다. LLM의 학습 데이터에 의존하지 않고, 실시간으로 관련 정보를 검색하여 프롬프트에 주입함으로써 사실 기반(fact-grounded) 출력을 보장합니다.

```
Query → Query Understanding → Search Knowledge Source → Retrieve Relevant Info → Augment Prompt → LLM Response
```

## 아키텍처 흐름

```
[사용자 질의]
     │
     ▼
[Query Understanding] ← 질의 분석, 의도 파악, 검색 쿼리 생성
     │
     ▼
[Retrieval Engine] ← 벡터 데이터베이스 / 키워드 인덱스 / 하이브리드 검색
     │
     ├── Dense Retrieval (의미 기반 벡터 유사도)
     ├── Sparse Retrieval (BM25/키워드 매칭)
     └── Re-ranking (관련성 재정렬)
     │
     ▼
[Context Augmentation] ← 검색 결과를 프롬프트에 주입
     │
     ▼
[LLM Generation] ← 증강된 컨텍스트 기반 응답 생성
     │
     ▼
[Response + Citation]
```

## 핵심 구성요소

### 1. Query Understanding (질의 이해)

| 구성 | 설명 |
|------|------|
| **의도 분류** | 질문 유형 파악 (사실 확인, 비교, 요약 등) |
| **쿼리 변환** | 검색에 최적화된 형태로 쿼리 재구성 |
| **쿼리 확장** | 동의어, 관련 용어를 추가하여 검색 범위 확대 |
| **다중 쿼리 생성** | 하나의 질문을 여러 검색 쿼리로 분해 |

### 2. Retrieval Strategy (검색 전략)

| 전략 | 방식 | 강점 | 약점 |
|------|------|------|------|
| **Dense Retrieval** | 임베딩 벡터 유사도 검색 | 의미적 유사성 포착 | 키워드 정확 매칭 약함 |
| **Sparse Retrieval** | BM25/TF-IDF 키워드 매칭 | 정확한 키워드 매칭 | 의미적 유사성 놓침 |
| **Hybrid** | Dense + Sparse 결합 | 두 방식의 장점 결합 | 가중치 튜닝 필요 |
| **Re-ranking** | 초기 검색 결과 재정렬 | 최종 관련성 향상 | 추가 지연 시간 |

### 3. Context Augmentation (컨텍스트 증강)

- 검색된 청크를 프롬프트의 컨텍스트 섹션에 삽입
- 출처 정보(문서명, 페이지, 날짜)를 함께 포함
- 컨텍스트 윈도우 한계 내에서 가장 관련성 높은 청크 우선 배치

### 4. Response Generation (응답 생성)

- 검색된 정보만을 기반으로 답변 생성 (환각 방지)
- 출처 인용(citation) 포함
- 검색 결과가 불충분할 경우 명시적으로 알림

## Description (단계별)

### Step 1: 문서 인덱싱 (오프라인 단계)

```
[원본 문서들]
     │
     ▼
[문서 로딩] → PDF, HTML, Markdown, DB 등 다양한 소스
     │
     ▼
[청킹(Chunking)] → 문서를 검색 가능한 단위로 분할
     │
     ▼
[임베딩 생성] → 각 청크를 벡터로 변환
     │
     ▼
[벡터 데이터베이스 저장] → 인덱싱 및 메타데이터 저장
```

#### 청킹 전략 비교

| 전략 | 방식 | 적합 상황 |
|------|------|----------|
| **고정 크기 청킹** | 일정 토큰/문자 수로 분할 | 균일한 문서, 빠른 구현 |
| **의미 기반 청킹** | 의미 단위(문단, 주제)로 분할 | 복잡한 문서, 높은 검색 품질 |
| **문서 구조 기반** | 헤딩, 섹션 등 구조 활용 | 잘 구조화된 문서 (기술 문서, 매뉴얼) |
| **슬라이딩 윈도우** | 겹치는 영역을 두고 분할 | 문맥 손실 최소화 |

### Step 2: 쿼리 처리 (온라인 단계)

```
[사용자 질의] → "분기별 매출 추이와 주요 변동 원인은?"
     │
     ▼
[쿼리 분석]
  ├── 의도: 데이터 분석 + 원인 설명
  ├── 키워드: 분기별, 매출, 추이, 변동, 원인
  └── 검색 쿼리 생성:
       ├── Query 1: "분기별 매출 실적 추이"
       └── Query 2: "매출 변동 원인 분석"
```

### Step 3: 검색 및 검색 결과 처리

```
[검색 쿼리]
     │
     ├── [Dense Search] → Top-K 벡터 유사도 결과
     ├── [Sparse Search] → Top-K BM25 결과
     │
     ▼
[결과 병합 및 중복 제거]
     │
     ▼
[Re-ranking] → 관련성 점수 기반 재정렬
     │
     ▼
[Top-N 최종 청크 선정]
```

### Step 4: 프롬프트 구성 및 응답 생성

```
System: "다음 참고 자료를 기반으로 사용자 질문에 답변하세요.
         참고 자료에 없는 내용은 '확인할 수 없습니다'라고 답하세요.
         답변 시 출처를 [1], [2] 형태로 인용하세요."

Context:
  [1] 2024년 Q1 매출 보고서 - "1분기 매출은 전년 대비 15% 증가..."
  [2] 경영진 분석 보고서 - "신규 시장 진출이 매출 증가의 주요 원인..."
  [3] ...

User: "분기별 매출 추이와 주요 변동 원인은?"
```

## 역량 (Capabilities)

| 역량 | 설명 |
|------|------|
| **사실 기반 출력** | 검색된 실제 데이터를 기반으로 응답하여 환각 최소화 |
| **지식 확장** | 파인튜닝 없이 외부 지식 소스 연결로 LLM 지식 범위 확장 |
| **동적 컨텍스트** | 지식 소스 업데이트만으로 최신 정보 반영 가능 |
| **출처 추적** | 응답의 근거를 명확한 출처와 함께 제시 |
| **도메인 특화** | 기업 내부 문서, 규정 등 특정 도메인 지식 활용 |

## 적합 사례

| 사례 | 설명 |
|------|------|
| **기업 지식 어시스턴트** | 사내 문서, 위키, 규정을 검색하여 직원 질문에 답변 |
| **컴플라이언스 봇** | 규정, 법률 문서를 기반으로 준수 여부 확인 및 가이드 |
| **고객 지원 Agent** | FAQ, 매뉴얼, 이전 티켓 기반으로 고객 문의 응대 |
| **문서 분석 Agent** | 대량 문서에서 특정 정보 검색 및 요약 |
| **연구 보조 Agent** | 논문, 보고서를 검색하여 연구 질문에 답변 |

## 구현 가이드

### 벡터 데이터베이스 선택

| 고려 사항 | 설명 |
|----------|------|
| **확장성** | 데이터 규모 증가에 따른 성능 유지 |
| **검색 속도** | 실시간 응답에 필요한 지연 시간 |
| **필터링** | 메타데이터 기반 사전 필터링 지원 여부 |
| **관리 용이성** | 인덱스 업데이트, 백업, 모니터링 |

### 임베딩 모델 선택 기준

| 기준 | 설명 |
|------|------|
| **언어 지원** | 한국어 등 대상 언어의 임베딩 품질 |
| **차원 수** | 높을수록 정보량 증가, 저장/검색 비용도 증가 |
| **도메인 적합성** | 범용 vs 도메인 특화 모델 |
| **비용** | API 호출 비용 vs 자체 호스팅 비용 |

### 청크 크기 최적화

```
[너무 작은 청크]          [적절한 청크 크기]          [너무 큰 청크]
- 문맥 부족              - 충분한 문맥 포함           - 노이즈 증가
- 검색 정확도 저하         - 검색 품질 양호            - 관련 없는 내용 포함
- 많은 청크 필요          - 적절한 청크 수            - 컨텍스트 윈도우 낭비

권장: 200~500 토큰, 오버랩 10~20%
(도메인과 문서 특성에 따라 조정)
```

### 관련성 점수 기준

```
[검색 결과 처리]
     │
     ├── 유사도 임계값 설정 (예: 0.7 이상)
     ├── 최대 청크 수 제한 (예: Top-5)
     ├── 메타데이터 필터 적용 (날짜, 카테고리, 권한)
     └── 다양성 확보 (MMR: Maximal Marginal Relevance)
```

## Agent 구성 예시

```
RAG Agent
├── System Prompt
│   ├── 역할 정의 (지식 기반 답변 전문가)
│   ├── 답변 규칙 (출처 인용, 불확실성 표현)
│   └── 컨텍스트 활용 가이드
├── Tools
│   ├── search_knowledge_base: 벡터 DB 검색
│   ├── keyword_search: 키워드 기반 검색
│   └── get_document_metadata: 문서 메타정보 조회
├── Retrieval Pipeline
│   ├── Query Processor
│   ├── Retriever (Dense + Sparse)
│   ├── Re-ranker
│   └── Context Builder
└── Response Policy
    ├── 반드시 출처 인용
    ├── 검색 결과 없으면 "확인 불가" 명시
    └── 추측성 답변 금지
```

## 고급 패턴

### Agentic RAG

기본 RAG에 Agent의 판단력을 추가한 패턴입니다:

```
[질의]
  │
  ▼
[Agent 판단] ← "이 질문에 답하려면 어떤 검색이 필요한가?"
  │
  ├── 검색 쿼리 자동 생성
  ├── 검색 결과 평가 → 부족하면 추가 검색
  ├── 여러 소스 교차 검증
  └── 최종 답변 합성
```

### Multi-step RAG

```
Step 1: 초기 검색 → 결과 분석
Step 2: 부족한 정보 식별 → 추가 검색 쿼리 생성
Step 3: 보완 검색 → 결과 통합
Step 4: 종합 답변 생성
```

### Self-correcting RAG

```
[응답 생성] → [자가 검증] → 검증 실패 시 → [재검색 + 재생성]
                │
                └── 검증 기준: 출처 일치, 논리적 일관성, 완전성
```

## 주의사항

1. **검색 품질이 출력 품질 결정**: 검색이 잘못되면 응답도 부정확 — 검색 파이프라인에 충분한 투자 필요
2. **청크 크기 트레이드오프**: 너무 작으면 문맥 부족, 너무 크면 노이즈 증가 — 도메인별 실험 필수
3. **컨텍스트 윈도우 한계**: 검색 결과가 많아도 LLM이 처리할 수 있는 토큰 수에 제한 — 우선순위 기반 선택
4. **임베딩 모델 일관성**: 인덱싱과 검색 시 동일한 임베딩 모델 사용 필수
5. **지식 소스 최신성**: 문서 업데이트 시 인덱스도 동기화해야 함 — 자동 재인덱싱 파이프라인 구축
6. **환각 완전 제거 불가**: RAG가 환각을 줄이지만, LLM이 검색 결과를 왜곡할 수 있음 — 출처 검증 메커니즘 추가
7. **보안 및 접근 제어**: 검색 결과에 사용자 권한 기반 필터링 적용 — 민감 문서 노출 방지
8. **비용 관리**: 임베딩 생성, 벡터 DB 호스팅, LLM 호출 비용을 종합적으로 고려
