# Observer Agent (관찰 및 모니터링 에이전트)

## 개념

Observer Agent는 시스템을 수동적으로 관찰하고, 패턴을 감지하며, 인사이트를 생성하고, 필요 시 액션을 트리거하는 Agent 패턴입니다. 직접 시스템을 변경하지 않고(non-invasive), 텔레메트리 데이터(로그, 메트릭, 이벤트)를 수집하여 AI 기반 분석을 수행합니다. 기존 규칙 기반 모니터링으로는 감지하기 어려운 복합 패턴과 이상 징후를 LLM의 추론 능력으로 발견합니다.

```
Ingest Telemetry → Parse Context → LLM Reasoning (Anomaly Detection, Trend Analysis) → Classify/Alert → Log to Memory
```

## 아키텍처 흐름

```
[데이터 소스]
  ├── 로그 (애플리케이션, 시스템, 접근 로그)
  ├── 메트릭 (CPU, 메모리, 응답 시간, 에러율)
  ├── 이벤트 (배포, 설정 변경, 사용자 활동)
  └── 트레이스 (분산 추적, 요청 흐름)
     │
     ▼
[이벤트 수집 레이어]
  ├── 스트림 수집 (실시간)
  ├── 배치 수집 (주기적)
  └── 폴링 (온디맨드)
     │
     ▼
[전처리 및 구조화]
  ├── 타임스탬프 정규화
  ├── 신원 식별 (서비스, 인스턴스, 사용자)
  ├── 트레이스 ID 연결
  └── 노이즈 필터링
     │
     ▼
[LLM 기반 추론 엔진]
  ├── 이상 탐지 (Anomaly Detection)
  ├── 추세 분석 (Trend Summarization)
  ├── 근본 원인 분석 (Root Cause Analysis)
  └── 상관관계 분석 (Correlation)
     │
     ▼
[분류 및 알림]
  ├── 심각도 분류 (Critical / Warning / Info)
  ├── 알림 전송 (메시지, 이메일, 티켓)
  └── 자동 대응 트리거 (선택적)
     │
     ▼
[메모리/로깅]
  ├── 분석 이력 저장
  ├── 패턴 데이터베이스 업데이트
  └── 대시보드 데이터 갱신
```

## 핵심 구성요소

### 1. Event Ingestion (이벤트 수집)

| 수집 방식 | 설명 | 적합 상황 |
|----------|------|----------|
| **스트림 수집** | 실시간 데이터 스트림 소비 | 즉각적 이상 탐지 필요 |
| **배치 수집** | 일정 주기로 데이터 수집 | 추세 분석, 일일/주간 리포트 |
| **이벤트 구독** | 이벤트 버스에서 특정 이벤트 구독 | 특정 이벤트 반응 (배포, 장애) |
| **폴링** | 주기적으로 데이터 소스 조회 | API 기반 외부 시스템 모니터링 |

### 2. Context Parsing (컨텍스트 파싱)

```
[원본 로그]
"2025-03-22 14:30:15 ERROR [order-service] [trace-id:abc123] Payment processing failed for order ORD-789. Timeout after 30s."

     │
     ▼
[구조화된 이벤트]
{
  "timestamp": "2025-03-22T14:30:15Z",
  "level": "ERROR",
  "service": "order-service",
  "trace_id": "abc123",
  "message": "Payment processing failed",
  "context": {
    "order_id": "ORD-789",
    "failure_reason": "timeout",
    "timeout_duration": "30s"
  }
}
```

| 파싱 항목 | 설명 |
|----------|------|
| **타임스탬프** | 이벤트 발생 시각 정규화 |
| **신원 (Identity)** | 서비스명, 인스턴스 ID, 사용자 |
| **트레이스 ID** | 분산 추적을 위한 요청 식별자 |
| **심각도** | 로그 레벨 (ERROR, WARN, INFO) |
| **컨텍스트** | 도메인 특화 정보 추출 |

### 3. LLM-based Reasoning (AI 추론 엔진)

| 추론 유형 | 설명 | 입력 예시 |
|----------|------|----------|
| **이상 탐지** | 정상 패턴과 다른 행동 감지 | 에러율 급증, 응답 시간 변화 |
| **추세 분석** | 시간에 따른 패턴 변화 요약 | 주간 트래픽 추이, 성능 변화 |
| **근본 원인 분석** | 문제의 근본 원인 추론 | 장애 발생 시 관련 이벤트 연결 |
| **상관관계 분석** | 여러 시그널 간의 관계 발견 | 배포 이벤트와 에러율 증가의 연관성 |
| **예측** | 현재 추세 기반 미래 상태 예측 | 리소스 고갈 시점, 장애 가능성 |

### 4. Classification and Alerting (분류 및 알림)

```
[LLM 분석 결과]
     │
     ▼
[심각도 분류]
  ├── Critical: 즉각 대응 필요 (서비스 중단, 보안 침해)
  ├── Warning: 주의 필요 (성능 저하, 에러율 증가 추세)
  ├── Info: 정보 제공 (추세 요약, 패턴 변화)
  └── Suppressed: 노이즈로 판단, 알림 안 함
     │
     ▼
[알림 채널 라우팅]
  ├── Critical → 즉시 메시지 + 전화 + 티켓 자동 생성
  ├── Warning → 메시지 알림 + 대시보드 표시
  ├── Info → 대시보드 표시 + 일일 리포트에 포함
  └── Suppressed → 로그만 기록
```

### 5. Memory and Logging (메모리/로깅)

```
[분석 이력 저장]
  ├── 감지된 패턴 데이터베이스
  │   └── "order-service에서 매주 월요일 09:00 트래픽 급증 패턴"
  ├── 알림 이력
  │   └── 과거 알림과 대응 결과 기록
  ├── 오탐 기록
  │   └── 잘못된 알림 → 향후 필터링에 활용
  └── 인사이트 아카이브
      └── 주간/월간 분석 리포트
```

## Description (단계별)

### Step 1: 데이터 수집 및 전처리

```
[데이터 소스들]
  ├── 애플리케이션 로그 → 로그 수집기
  ├── 인프라 메트릭 → 메트릭 수집기
  ├── 배포 이벤트 → 이벤트 버스 구독
  └── 사용자 활동 → 활동 로그 수집
     │
     ▼
[전처리]
  ├── 구조화: 비정형 로그 → JSON 변환
  ├── 정규화: 타임스탬프, 서비스명 통일
  ├── 필터링: 노이즈 제거 (health check, debug 로그)
  ├── 집계: 윈도우 기반 메트릭 집계 (1분/5분/1시간)
  └── 보강: 서비스 메타데이터 추가 (팀, 환경, 버전)
```

### Step 2: 분석 트리거 및 LLM 추론

```
[트리거 조건]
  ├── 임계값 기반: 에러율 > 5%, 응답시간 > 2초
  ├── 패턴 기반: 30분 내 동일 에러 10회 이상
  ├── 이벤트 기반: 배포 완료 후 15분
  └── 주기 기반: 매 시간 추세 분석
     │
     ▼
[LLM에 전달할 컨텍스트 구성]

Prompt 예시:
"다음은 최근 30분간의 order-service 모니터링 데이터입니다.

=== 메트릭 ===
- 에러율: 2% → 8% (15분간 4배 증가)
- 평균 응답시간: 200ms → 1500ms
- 요청량: 분당 500건 (정상 범위)

=== 최근 에러 로그 (샘플) ===
[14:20] Payment timeout after 30s (5건)
[14:25] Payment timeout after 30s (12건)
[14:30] Database connection pool exhausted (3건)

=== 최근 이벤트 ===
- 14:00 payment-service v2.3.1 배포
- 14:15 DB 마이그레이션 스크립트 실행

분석해주세요: 이상 여부, 가능한 원인, 권장 조치"
```

### Step 3: 분류 및 알림

```
[LLM 분석 결과]
{
  "anomaly_detected": true,
  "severity": "critical",
  "summary": "payment-service 배포(v2.3.1) 후 결제 타임아웃 급증",
  "root_cause_hypothesis": [
    "payment-service v2.3.1 배포에서 DB 커넥션 풀 설정 변경 가능성",
    "DB 마이그레이션이 테이블 락을 유발했을 가능성"
  ],
  "recommended_actions": [
    "payment-service v2.3.1 롤백 고려",
    "DB 커넥션 풀 상태 확인",
    "마이그레이션 락 상태 확인"
  ],
  "affected_services": ["order-service", "payment-service"],
  "correlation": "배포 시점(14:00)과 에러 증가 시점(14:20) 사이 20분 지연 — 점진적 커넥션 풀 고갈 패턴"
}
     │
     ▼
[알림 전송]
  ├── 메시지: "CRITICAL: payment-service 배포 후 결제 실패율 급증..."
  ├── 담당 팀: payment-team, infra-team
  └── 티켓 자동 생성: INC-2025-0322-001
```

### Step 4: 메모리 저장 및 학습

```
[패턴 데이터베이스 업데이트]
  {
    "pattern_id": "pat-042",
    "type": "post_deployment_failure",
    "description": "배포 후 DB 커넥션 풀 고갈로 인한 타임아웃",
    "indicators": ["에러율 급증", "응답시간 증가", "커넥션 풀 소진"],
    "correlation_window": "배포 후 15-30분",
    "resolution": "롤백 또는 커넥션 풀 설정 조정"
  }
```

## 역량 (Capabilities)

| 역량 | 설명 |
|------|------|
| **수동적/비침습적** | 시스템을 변경하지 않고 관찰만 수행하여 안전 |
| **높은 확장성** | 다양한 데이터 소스를 동시에 모니터링 가능 |
| **AI 기반 상관관계 분석** | 규칙 기반으로는 감지 어려운 복합 패턴 발견 |
| **자연어 인사이트** | 기술적 데이터를 이해하기 쉬운 자연어 분석으로 변환 |
| **감사 및 컴플라이언스** | 모든 관찰 및 분석 기록을 자동 보존 |
| **적응적 학습** | 과거 패턴과 오탐 기록을 학습하여 정확도 향상 |

## 적합 사례

| 사례 | 설명 |
|------|------|
| **마이크로서비스 관측성** | 분산 서비스 간의 복합 장애 패턴 탐지 및 근본 원인 분석 |
| **모델 드리프트 모니터링** | ML 모델의 입력 분포 변화, 성능 저하 감지 |
| **고객 활동 분석** | 사용자 행동 패턴 변화 감지, 이탈 예측, 이상 사용 탐지 |
| **코드 리뷰 모니터링** | PR 패턴 분석, 리뷰 병목 감지, 코드 품질 추세 파악 |
| **보안 로그 모니터링** | 비정상 접근 패턴, 인증 실패 급증, 권한 상승 시도 감지 |
| **인프라 용량 계획** | 리소스 사용 추세 분석, 용량 부족 예측, 최적화 권고 |

## 구현 가이드

### 데이터 수집 레이어 설계

```
Observer Agent 수집 아키텍처

[로그 소스]                    [메트릭 소스]
  │                              │
  ▼                              ▼
[로그 수집기]                  [메트릭 수집기]
  │                              │
  └──────────┬───────────────────┘
             │
             ▼
     [메시지 큐 / 버퍼]
             │
             ▼
     [전처리 서버리스 함수]
       ├── 구조화
       ├── 필터링
       └── 집계
             │
             ▼
     [분석 트리거]
       ├── 임계값 기반
       ├── 패턴 기반
       └── 주기 기반
             │
             ▼
     [Observer Agent (LLM)]
             │
             ▼
     [알림 / 저장]
```

### 전처리 및 구조화 전략

| 단계 | 처리 내용 | 목적 |
|------|----------|------|
| **파싱** | 비정형 로그 → 구조화된 JSON | LLM이 이해할 수 있는 형태로 변환 |
| **필터링** | 노이즈 제거 (health check 등) | 분석 비용 절감, 정확도 향상 |
| **집계** | 시간 윈도우 기반 통계 | 추세 파악, 이상치 탐지 |
| **보강** | 서비스 메타데이터 추가 | 컨텍스트 풍부화 |
| **샘플링** | 대량 데이터 중 대표 샘플 추출 | LLM 토큰 비용 관리 |

### 추론 엔진 설계

```
[분석 유형별 프롬프트 전략]

1. 이상 탐지 프롬프트:
   "다음 메트릭 데이터에서 정상 범위를 벗어난 항목을 식별하세요.
    과거 7일 평균과 현재 값을 비교하세요."

2. 근본 원인 분석 프롬프트:
   "장애 타임라인과 관련 이벤트를 분석하여
    가장 가능성 높은 원인을 순서대로 나열하세요."

3. 추세 요약 프롬프트:
   "지난 1주일간의 서비스 메트릭 추이를 요약하고,
    주목할 만한 변화와 잠재적 위험을 설명하세요."

4. 보안 분석 프롬프트:
   "다음 인증 로그에서 비정상적 패턴을 감지하세요.
    브루트포스, 계정 탈취, 내부 위협 징후를 확인하세요."
```

### 알림 및 에스컬레이션 설계

```
[알림 정책]

Level 1 (자동 처리):
  ├── 알려진 패턴 → 자동 대응 트리거
  └── 예: 디스크 사용량 90% → 자동 정리 스크립트

Level 2 (담당자 알림):
  ├── 경고 수준 이상 → 담당 팀에 알림
  └── 예: 에러율 증가 → 개발팀 메시지

Level 3 (에스컬레이션):
  ├── 미대응 30분 초과 → 상위 관리자 알림
  └── 예: Critical 장애 미대응 → 인시던트 커맨더 호출

Level 4 (자동 완화):
  ├── 사전 정의된 완화 조치 자동 실행
  └── 예: 트래픽 급증 → 오토스케일링 트리거
```

### Agent 구성 예시

```
Observer Agent
├── System Prompt
│   ├── 역할: 시스템 관측 및 분석 전문가
│   ├── 분석 원칙: 객관적 데이터 기반, 가설 검증
│   └── 알림 기준: 심각도별 분류 규칙
├── Ingestion Tools
│   ├── query_logs: 로그 검색 및 조회
│   ├── query_metrics: 메트릭 데이터 조회
│   ├── get_events: 이벤트 히스토리 조회
│   └── get_traces: 분산 추적 데이터 조회
├── Analysis Tools
│   ├── detect_anomaly: 통계적 이상 탐지
│   ├── correlate_events: 이벤트 간 상관관계 분석
│   └── compare_baseline: 기준선 대비 비교
├── Action Tools
│   ├── send_alert: 알림 전송
│   ├── create_ticket: 인시던트 티켓 생성
│   └── update_dashboard: 대시보드 갱신
└── Memory
    ├── pattern_db: 알려진 패턴 데이터베이스
    ├── alert_history: 알림 이력
    └── false_positive_log: 오탐 기록
```

## 고급 패턴

### Proactive Observer (선제적 관찰자)

```
[현재 추세 분석]
     │
     ▼
[미래 예측]
  ├── "현재 디스크 증가율로 72시간 후 용량 부족 예상"
  ├── "트래픽 패턴상 금요일 피크 시 응답시간 SLA 위반 가능"
  └── "인증서 만료일 7일 전 알림"
     │
     ▼
[선제적 권고]
  └── "디스크 확장 또는 로그 로테이션 정책 조정 권장"
```

### Multi-layer Observer (다계층 관찰자)

```
[Layer 1] 인프라 Observer → CPU, 메모리, 네트워크
[Layer 2] 애플리케이션 Observer → 에러율, 응답시간, 처리량
[Layer 3] 비즈니스 Observer → 주문량, 전환율, 매출
     │
     ▼
[Cross-layer Correlator]
  └── "인프라 CPU 급증 → 애플리케이션 응답 지연 → 비즈니스 전환율 하락" 연결
```

### Observer with Feedback Loop

```
[관찰 → 분석 → 알림 → 대응]
                              │
                              ▼
                        [대응 결과 관찰]
                              │
                              ▼
                        [효과 분석]
                              │
                              ├── 해결됨 → 패턴 DB에 성공 사례 기록
                              └── 미해결 → 에스컬레이션 + 추가 분석
```

## 주의사항

1. **대용량 데이터 처리**: 로그/메트릭 데이터는 매우 방대함 — 샘플링, 집계, 필터링으로 LLM에 전달할 데이터 최적화
2. **오탐 관리 (False Positives)**: AI 분석이 잘못된 알림을 생성할 수 있음 — 오탐 피드백 루프로 지속적 정확도 개선
3. **지연 시간 요구사항**: 실시간 모니터링 시 LLM 호출 지연이 문제 — 임계값 기반 사전 필터링 후 LLM 분석
4. **LLM 호출 비용**: 지속적 모니터링은 대량의 LLM 호출 발생 — 비용 효율적 모델 선택, 배치 분석, 캐싱
5. **알림 피로 (Alert Fatigue)**: 너무 많은 알림은 무시됨 — 중복 제거, 그룹화, 심각도 기반 필터링
6. **데이터 보안**: 모니터링 데이터에 민감 정보 포함 가능 — PII 마스킹, 접근 제어, 데이터 보존 정책
7. **관찰자 효과**: Observer Agent 자체가 시스템 부하를 발생시킬 수 있음 — 경량 수집, 비동기 처리, 리소스 제한
8. **컨텍스트 부족**: 단편적 데이터만으로는 정확한 분석이 어려움 — 충분한 시간 범위와 관련 데이터 소스 포함
