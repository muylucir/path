# AI-Assisted Workflow 설계 가이드

자율성 요구도가 낮은(≤5) 프로젝트를 위한 설계 가이드입니다.
전체 흐름은 결정적 파이프라인이 제어하고, 특정 단계에서만 AI(LLM)를 활용합니다.

## 파이프라인 vs 에이전트 비교

| 항목 | AI-Assisted Workflow (파이프라인) | Agentic AI (에이전트) |
|------|:-------------------------------:|:--------------------:|
| 흐름 제어 | 코드/워크플로우 엔진이 결정 | LLM이 동적으로 결정 |
| 예측 가능성 | 높음 (결정적) | 낮음 (확률적) |
| 디버깅 | 용이 (단계별 추적) | 어려움 (동적 경로) |
| 운영 리스크 | 낮음 | 높음 |
| 유연성 | 낮음 (사전 정의된 경로) | 높음 (동적 대응) |
| 비용 예측 | 용이 (고정 LLM 호출 수) | 어려움 (가변 호출 수) |
| 적합 상황 | 명확한 규칙, 구조화된 입출력 | 예측 불가 입력, 동적 판단 |

## 워크플로우 엔진 선택 기준

| 엔진 | 적합 상황 | 장점 | 단점 |
|------|----------|------|------|
| **AWS Step Functions** | AWS 네이티브, 서버리스 | 시각적 편집, 에러 핸들링 내장, Bedrock 직접 연동 | AWS 종속 |
| **Amazon EventBridge** | 이벤트 기반 트리거 | 느슨한 결합, 다양한 소스 | 복잡한 순차 흐름에 부적합 |
| **Apache Airflow (MWAA)** | 데이터 파이프라인, 배치 처리 | DAG 기반, 스케줄링, 모니터링 | 실시간 처리에 부적합 |
| **코드 기반 (Python/TS)** | 단순 파이프라인, 프로토타입 | 빠른 구현, 유연성 | 모니터링/재시도 직접 구현 |

## AI 통합 포인트 설계

파이프라인 내에서 AI를 활용하는 주요 단계:

### 1. 분류/라우팅 단계
- 입력 데이터를 카테고리로 분류하여 다음 경로 결정
- 예: 고객 문의 → {환불, 배송, 기술지원} 분류

### 2. 추출/변환 단계
- 비정형 데이터에서 구조화된 정보 추출
- 예: 이메일 본문 → JSON 구조 추출

### 3. 생성/요약 단계
- 수집된 데이터를 바탕으로 텍스트 생성
- 예: 분석 결과 → 요약 보고서 생성

### 4. 검증/평가 단계
- 이전 단계 결과의 품질 검증
- 예: 생성된 응답의 정확성/적절성 평가

## 각 단계의 입출력 스키마 설계

각 AI 통합 포인트는 명확한 입출력 스키마를 정의해야 합니다:

```
[Stage N] ──입력 스키마──▶ [AI 호출] ──출력 스키마──▶ [Stage N+1]
```

- **입력**: 이전 단계의 출력 + 컨텍스트 (JSON/텍스트)
- **프롬프트**: 태스크 지시 + 출력 포맷 지정
- **출력**: 구조화된 JSON (Structured Output 권장)
- **폴백**: AI 실패 시 기본값 또는 사람에게 에스컬레이션
