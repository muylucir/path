# P.A.T.H Agent Designer

> AI Agent 아이디어를 검증하고, 개발팀이 바로 착수할 수 있는 구현 명세서를 자동 생성하는 도구

---

## P.A.T.H란?

**P.A.T.H** = **P**roblem → **A**ssessment → **T**echnical Review → **H**andoff

AI Agent를 도입하려면 "좋은 아이디어"만으로는 부족합니다. 그 아이디어가 실현 가능한지, 지금 바로 시작할 수 있는 상태인지, 어떤 구조로 만들어야 하는지를 먼저 확인해야 합니다.

P.A.T.H는 이 과정을 4단계로 안내합니다.

```
여러분의 아이디어
       ↓
  Step 1: Problem          — 문제를 구조화합니다
       ↓
  Step 2: Assessment       — 준비 상태를 점검합니다
       ↓
  Step 3: Technical Review — 구현 방식을 결정합니다
       ↓
  Step 4: Handoff          — 구현 명세서를 생성합니다
       ↓
  개발팀에 전달할 수 있는 문서
```

---

## 4단계 상세

### Step 1: Problem — 문제 구조화

"이 업무를 자동화하고 싶다"는 요구사항을, 기획팀과 개발팀이 같은 그림을 볼 수 있도록 정리합니다.

| 입력 항목 | 설명 | 예시 |
|----------|------|------|
| **Pain Point** | 자동화하고 싶은 업무의 구체적 문제 | 하루 100건 고객 이메일 답변에 2시간 소요 |
| **트리거 타입** | Agent가 언제 실행되는가 | Event-Driven, Scheduled, On-Demand, Streaming, Conditional |
| **수행 작업** | Agent가 하는 일 (복수 선택) | 정보 수집, 분석/이해, 추론/판단, 생성/작성, 검증/반복 |
| **산출물** | 최종 결과물 (복수 선택) | 텍스트 응답, 문서/보고서, 구조화된 데이터, 외부 시스템 변경 |
| **Human-in-Loop** | 사람이 언제 개입하는가 | 완전 자동, 실행 전 승인 필요, 불확실할 때만 개입, AI와 함께 작업 |
| **오류 허용도** | AI 실수의 허용 범위 | 틀려도 괜찮음, 사람이 검토 후 실행, 높은 정확도 필요, 매우 높은 정확도 필요 |

추가로 Agent가 활용할 데이터 소스(데이터베이스, API, MCP 서버 등)와 업무 맥락도 자유 텍스트로 입력할 수 있습니다.

**이 단계를 거치면**: 막연했던 요구사항이 구체적인 문제 정의로 바뀝니다. 기획팀이 "이걸 자동화하고 싶어요"라고 말한 것을, 개발팀이 이해할 수 있는 형태로 변환합니다.

---

### Step 2: Assessment — 준비도 점검

아이디어가 좋아도, 데이터가 없거나 시스템 연동이 어려우면 프로젝트가 중간에 멈춥니다. AI가 5가지 핵심 항목을 점검하고, 자율성 요구도를 별도로 평가합니다.

#### 준비도 점검 항목 (5개, 각 10점)

| 항목 | 준비됨 (8-10점) | 양호 (6-7점) | 보완 필요 (4-5점) | 준비 필요 (0-3점) |
|------|----------------|-------------|-----------------|-----------------|
| **데이터 접근성** | API로 바로 가져올 수 있음 | 파일 기반 접근 (CSV, Excel) | DB 직접 접근 필요 | 수기 데이터만 존재 |
| **판단 명확성** | 명확한 규칙이 정리되어 있음 | 경험적으로는 알지만 문서화 안 됨 | 일부 기준만 존재 | 담당자의 감에 의존 |
| **오류 허용도** | 틀려도 업무에 큰 지장 없음 | 대부분 맞아야 하지만 여유 있음 | 90% 이상 정확해야 함 | 반드시 100% 정확해야 함 |
| **지연 요구사항** | 몇 시간 걸려도 괜찮음 | 수 분 내 처리 가능 | 1분 이내 응답 필요 | 3초 이내 실시간 필요 |
| **통합 복잡도** | MCP/SDK로 바로 연동 가능 | API 존재하나 문서 부족 | 3~5개 시스템과 연결 필요 | API 없는 폐쇄 시스템 |

각 항목은 4단계로 평가됩니다: 준비됨, 양호, 보완 필요, 준비 필요. 항목별로 현재 상태와 평가 근거가 함께 제공됩니다.

#### 자율성 요구도 (별도 평가, 10점 만점)

준비도 점검과 별도로, 해당 업무에 **AI가 얼마나 자율적으로 판단해야 하는지**를 평가합니다. 이 점수는 50점 만점의 준비도 점수에 포함되지 않으며, 다음 단계(Step 3)에서 자동화 수준을 결정하는 데 활용됩니다.

| 점수 | 자율성 수준 | 예시 |
|------|-----------|------|
| **9-10점** | 완전 자율 | 예측 불가능한 상황에서 독립적 판단 필요 |
| **7-8점** | 높은 자율 | 상황에 따라 최적 경로를 스스로 선택 |
| **5-6점** | 보통 | 일부 판단 필요하지만 대부분 정해진 흐름 |
| **3-4점** | 낮은 자율 | 거의 결정적, AI가 특정 단계만 보조 |
| **0-2점** | 자율 불필요 | 완전히 결정적인 파이프라인 |

자율성 점수가 6점 이상이면 "Agentic AI 권장", 5점 이하이면 "AI-Assisted Workflow 가능"으로 표시됩니다.

#### 종합 판정 (총 50점)

| 점수 | 판정 | 의미 |
|------|------|------|
| **40~50점** | 바로 진행 | 프로토타입 개발을 바로 시작할 수 있습니다 |
| **30~39점** | 보완 후 진행 | 시작할 수 있지만 리스크 관리가 필요합니다 |
| **20~29점** | 재검토 권장 | 낮은 항목을 보완한 뒤 다시 점검하세요 |
| **20점 미만** | 준비 필요 | 현재 상태로는 어렵습니다. 다른 접근을 고려하세요 |

#### 재평가

점수가 낮은 항목(보완 필요, 준비 필요)에는 개선 계획을 직접 입력할 수 있습니다. 양호 항목에도 추가 개선사항을 선택적으로 입력할 수 있습니다. 개선 계획을 1개 이상 입력하면 "개선 방안 반영하여 재평가" 버튼이 활성화되어, 그 계획이 반영된 점수를 다시 받을 수 있습니다.

```
예시)
  데이터 접근성 4점
  → 개선 계획: "ERP 시스템에 조회 API를 개발할 예정"
  → 재평가: 7점으로 상향
```

또한, Step 1에서 미처 입력하지 못한 데이터 소스나 추가 컨텍스트를 이 단계에서 보완할 수 있으며, 입력한 내용은 다음 단계 분석에 자동으로 반영됩니다.

**이 단계를 거치면**: 개발 2주차에 발견할 문제("이 데이터는 접근이 안 됩니다", "이건 사람이 꼭 봐야 합니다")를 프로젝트 시작 전에 파악할 수 있습니다. 또한 자율성 요구도를 통해, AI Agent가 단순 보조 수준이면 되는지 자율 판단이 필요한지도 사전에 파악합니다.

---

### Step 3: Technical Review — 구현 방식 결정

준비도 결과와 자율성 요구도를 바탕으로, AI가 적합한 Agent 구조와 자동화 수준을 추천합니다. 이 단계는 **AI와의 대화 형태**로 진행됩니다.

#### 대화 기반 분석

AI가 먼저 준비도 점검 결과를 요약하고, 적합한 패턴을 제안합니다. 이후 질문에 대해 **객관식 선택지**(복수 선택 가능) 또는 **직접 입력**으로 답변할 수 있습니다. 자유 텍스트 입력도 가능하여, 추가적인 요구사항이나 질문을 전달할 수 있습니다.

- "우리 업무는 실시간 처리가 중요한데, 이 방식이 맞나요?"
- "Agent를 여러 개 쓰는 게 왜 필요한 건가요?"
- "AI-Assisted Workflow로도 충분하지 않을까요?"
- "더 단순한 방식으로는 안 되나요?"

충분히 논의한 뒤 **"패턴 확정"** 버튼을 클릭하면, 대화 내용을 기반으로 최종 분석 결과가 생성됩니다.

#### 자동화 수준 결정

Step 2에서 평가한 자율성 요구도를 기반으로, 적합한 자동화 수준을 결정합니다. 대화 과정에서 자율성 점수가 재조정될 수도 있습니다.

| 자동화 수준 | 자율성 점수 | 설명 |
|-----------|-----------|------|
| **AI-Assisted Workflow** | 5점 이하 | 결정적 파이프라인 + 특정 단계에서 AI 보조. 흐름이 정해져 있고, AI는 특정 작업(분류, 요약 등)만 담당 |
| **Agentic AI** | 6점 이상 | Agent가 자율적으로 도구를 선택하고 판단. 상황에 따라 다른 경로를 선택하며, 반복·수정 가능 |

#### Agent 구조 추천

| 구조 | 적합한 경우 | 예시 |
|------|-----------|------|
| **단계적 추론형** | 정보를 모아서 판단해야 하는 업무 | 고객 문의 분석 → 답변 생성 |
| **자가 검토형** | 결과물의 품질이 중요한 업무 | 보고서 작성 후 스스로 검토/수정 |
| **도구 활용형** | 외부 시스템을 조회하거나 실행해야 하는 업무 | DB 조회, API 호출, 파일 처리 |
| **계획 수립형** | 여러 단계를 순서대로 처리해야 하는 업무 | 다단계 승인 프로세스 |
| **다중 Agent 협업형** | 역할을 나눠야 할 만큼 복잡한 업무 | 분석 + 작성 + 검수를 각각 담당 |
| **사람 검토 포함형** | 최종 판단에 사람이 필요한 업무 | 법무/재무 검토가 포함된 프로세스 |

다중 Agent 협업이 필요한 경우, 구체적인 협업 패턴도 추천합니다:

| 협업 패턴 | 설명 |
|----------|------|
| **Agents as Tools** | Orchestrator가 전문 Agent를 도구처럼 호출 |
| **Swarm** | 동등한 Agent들이 handoff로 협업 |
| **Graph** | 방향성 그래프로 정보 흐름 정의 |
| **Workflow** | 미리 정의된 순서로 태스크 실행 |

**이 단계를 거치면**: "어떤 방식으로, 어떤 수준으로 만들 것인가"에 대한 기술적 결정이 내려지고, 그 이유도 함께 정리됩니다.

---

### Step 4: Handoff — 결과 확인 및 명세서 생성

Step 3에서 패턴이 확정되면, 분석 결과를 3개 탭에서 확인할 수 있습니다.

#### 분석 결과 탭

Step 1~3의 전체 분석 과정을 한눈에 정리합니다.

| 섹션 | 내용 |
|------|------|
| **입력 정보 요약** | Pain Point, 트리거, 수행 작업, 산출물, Human-in-Loop, 오류 허용도 |
| **준비도 점검 결과** | 5개 항목별 점수·근거, 개선 계획 반영 결과, 자율성 요구도 |
| **PROCESS (Claude 분석)** | AI가 분석한 세부 처리 단계 |
| **리스크 및 고려사항** | 식별된 위험 요소 |

이 탭에서 **인쇄 / PDF 저장** 기능을 통해 분석 결과를 문서로 내보낼 수 있습니다.

#### 대화 내역 탭

Step 3에서 AI와 나눈 대화 전체를 열람할 수 있습니다.

#### 명세서 탭

5개의 전문 AI가 명세서를 작성합니다. 1단계 완료 후 2~4단계를 병렬로 동시 실행하여 빠르게 결과를 생성합니다.

| 순서 | 작성 내용 | 실행 방식 |
|------|----------|----------|
| 1단계 | Agent 구조 설계 — 역할 분담, 처리 흐름, 상태 관리 | 순차 |
| 2단계 | 아키텍처 다이어그램 — 시각적 처리 흐름도 (Mermaid) | 병렬 실행 |
| 3단계 | 프롬프트 설계 — Agent별 프롬프트 엔지니어링 | 병렬 실행 |
| 4단계 | 도구 정의 — 필요한 도구 스키마 및 입출력 스펙 | 병렬 실행 |
| 5단계 | 최종 문서 통합 — 하나의 명세서로 조합 | 순차 |

```
1단계: 구조 설계  →  [2단계: 다이어그램]  →  5단계: 통합
                   [3단계: 프롬프트  ]
                   [4단계: 도구 정의 ]
                   (2~4단계 병렬 실행)
```

#### 명세서에 포함되는 내용

| 섹션 | 설명 |
|------|------|
| **요약** | 프로젝트 개요, 핵심 결정 사항, 자동화 수준 |
| **Agent 설계** | 선택된 구조와 선택 이유 |
| **역할 정의** | Agent별 담당 업무와 책임 범위 |
| **처리 흐름** | 입력부터 출력까지의 단계별 흐름 |
| **아키텍처 다이어그램** | 시각적 구조도 (Mermaid 다이어그램) |
| **프롬프트 설계** | Agent별 프롬프트 엔지니어링 가이드 |
| **도구 정의** | 필요한 도구 스키마 및 입출력 스펙 |

생성된 명세서는 **마크다운 파일로 다운로드**하거나, **세션으로 저장**하여 나중에 다시 확인할 수 있습니다. 필요하면 **재생성**도 가능합니다.

**이 단계를 거치면**: 기획서에 첨부하거나, 개발팀에 바로 전달하거나, 외부 개발사에 요구사항으로 보낼 수 있는 문서가 만들어집니다.

---

## 세션 관리

분석 결과를 저장하면, **세션 목록** 페이지에서 이전에 저장한 모든 분석을 확인하고 관리할 수 있습니다.

| 기능 | 설명 |
|------|------|
| **세션 목록 조회** | 날짜, Pain Point, 준비도 점수, 판정을 테이블에서 확인 |
| **세션 불러오기** | 이전 분석을 불러와서 결과를 다시 확인하거나 이어서 작업 |
| **세션 삭제** | 더 이상 필요하지 않은 세션을 삭제 |
| **새 분석** | 새로운 분석을 처음부터 시작 |
| **검색 및 정렬** | Pain Point 키워드 검색, 날짜·점수 기준 정렬 |

## 활용 시나리오

### 시나리오 1: 기획팀과 개발팀의 킥오프

**상황**: 기획팀이 "고객 문의 자동 분류"를 요청, 개발팀은 구체적인 요구사항을 모름

| | P.A.T.H 없이 | P.A.T.H 활용 |
|--|-------------|-------------|
| **시작** | "자동 분류해주세요" (막연) | P.A.T.H로 문제 구조화 |
| **중간** | 개발 2주차에 "분류 기준이 뭐죠?" | 킥오프에서 판단 명확성 5점 확인, 기준 정리 먼저 진행 |
| **결과** | 재작업, 일정 지연 | 준비된 상태에서 개발 시작 |

## 결과물 요약

| 결과물 | 누가, 어디에 쓰는가 |
|--------|-------------------|
| **준비도 점수 (50점 만점)** | 프로젝트 매니저 — Go/No-Go 판단 근거 |
| **자율성 요구도 (10점 만점)** | 기획팀/개발팀 — AI 자동화 수준 결정 근거 |
| **항목별 진단 + 개선 방향** | 기획팀 — 사전 준비 항목 도출 |

| **Agent 구조 추천 + 근거** | 개발팀 — 아키텍처 설계 출발점 (싱글/멀티 에이전트, 협업 패턴) |

| **세션 저장 및 관리** | 전체 팀 — 분석 이력 관리, 이전 결과 재활용 |

## P.A.T.H의 핵심 가치

```
  만들기 전에, 만들 수 있는지 확인한다.
  만들 수 있다면, 어떻게 만들어야 하는지 정리한다.
  정리가 끝나면, 바로 시작할 수 있는 문서를 만든다.
```

- 개발 착수 후 발견할 문제를 **사전에** 발견합니다
- 기획자와 개발팀이 **같은 언어**로 대화할 수 있게 합니다
- 아이디어에서 구현 계획까지의 **간극을 메웁니다**
