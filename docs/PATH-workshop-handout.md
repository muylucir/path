## 1. 워크샵 개요

### 워크샵 목표

| 단계 | 활동 | 산출물 |
|------|------|--------|
| **Step A** | PATH로 AI Agent 아이디어 검증 | 준비도 평가 + 패턴 분석 + 명세서 |
| **Step B** | AI-DLC로 코드 구현 | 동작 가능한 Agent 코드 |
| **Step C** | Mock 데이터로 테스트 | 실제 입출력 결과 확인 |

### 진행 방식

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   Guided 실습 (함께)          자유 주제 실습 (각자)   │
│   ┌───────────────┐          ┌───────────────┐      │
│   │ 사전 주제로    │    →     │ 본인 주제로    │      │
│   │ PATH 체험     │          │ PATH 실습     │      │
│   │ + 코드 구현    │          │ + 코드 구현    │      │
│   │ + Mock 테스트  │          │ + Mock 테스트  │      │
│   └───────────────┘          └───────────────┘      │
│                                                     │
│   "이렇게 하는 거구나"         "내 문제에 적용해보자"    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 2. P.A.T.H 프레임워크 소개

**P.A.T.H** = Problem → Technical → Agent Pattern → Handoff

| Step | 이름 | 하는 일 |
|------|------|---------|
| **P (Step 1)** | 기본 정보 입력 | 자동화하고 싶은 업무를 구조화하여 입력 |
| **T (Step 2)** | 준비도 점검 | AI가 5가지 기준으로 실현 가능성 평가 (0~50점) |
| **A (Step 3)** | 패턴 분석 | 적합한 Agent 아키텍처 패턴 추천 + 대화형 개선 |
| **H (Step 4)** | 명세서 생성 | 구현 가능한 수준의 상세 명세서 자동 생성 |

### 준비도 평가 기준 (Step 2)

| 기준 | 한국어 | 평가 내용 | 예시 |
|------|--------|----------|------|
| `data_access` | 데이터 접근성 | 필요한 데이터에 접근할 수 있는가? | API 존재 여부, 데이터 포맷 |
| `decision_clarity` | 판단 명확성 | 자동화할 판단 기준이 명확한가? | 분류 기준, 규칙 정의 수준 |
| `error_tolerance` | 오류 허용도 | 오류가 발생했을 때 감당 가능한가? | 재무 vs 내부 참고용 |
| `latency` | 지연 요구사항 | 얼마나 빠르게 처리해야 하는가? | 실시간 vs 배치 |
| `integration` | 통합 복잡도 | 외부 시스템 연동이 복잡한가? | 연동 시스템 수, API 표준화 |

### 준비도 레벨

| 레벨 | 아이콘 | 총점 기준 | 의미 |
|------|--------|----------|------|
| READY | ✅ | 40점 이상 | 바로 진행 가능 |
| GOOD | 🔵 | 30점 이상 | 약간의 보완으로 충분 |
| NEEDS_WORK | 🟡 | 20점 이상 | 추가 준비 권장 |
| PREPARE | 🟠 | 20점 미만 | 상당한 준비 필요 |

## 3. Step 1 입력 가이드 — 무엇을, 왜 선택하는가

Step 1은 PATH의 출발점입니다. 여기서 입력한 내용이 이후 모든 분석의 기반이 되므로, 각 필드의 의미를 정확히 이해하고 선택하는 것이 중요합니다.

### 3.1 painPoint (자유 입력)

> "자동화하고 싶은 업무의 현재 고통"을 구체적으로 서술합니다.

**잘 쓴 예시:**
> 하루 평균 200건의 고객 문의 이메일이 들어오는데, 담당자가 수동으로 유형을 분류하고 초안을 작성하느라 하루 3시간 이상 소요. 긴급 문의 대응이 늦어져 고객 불만 증가

**아쉬운 예시:**
> 고객 문의 처리가 힘듭니다

**작성 팁:**
- **숫자**를 넣으세요 (건수, 시간, 인원)
- **현재 방식**이 왜 문제인지 쓰세요
- **결과적 피해**가 뭔지 쓰세요

---

### 3.2 inputType (단일 선택)

> "이 업무는 언제 시작되는가?"를 기준으로 선택합니다.

| 선택지 | 판단 기준 | 전형적 케이스 |
|--------|----------|-------------|
| **Event-Driven (이벤트 발생 시)** | 무언가가 발생했을 때 자동 시작 | 이메일 도착, 주문 접수, 알림 수신 |
| **Scheduled (정해진 시간)** | 주기적으로 반복 실행 | 매일 아침, 매주 금요일, 매월 1일 |
| **On-Demand (사용자 요청)** | 사람이 필요할 때 직접 요청 | 챗봇 질문, 검색, 버튼 클릭 |
| **Streaming (실시간 스트림)** | 데이터가 계속 흘러들어옴 | 로그 모니터링, SNS 피드, IoT 센서 |
| **Conditional (조건 충족 시)** | 특정 조건을 만족할 때만 실행 | 재고 부족 시, 이상 탐지 시 |

**핵심 질문: "누가/무엇이 이 업무를 시작하는가?"**

```
사람이 시작      → On-Demand
시스템이 감지    → Event-Driven 또는 Conditional
시계가 알려줌    → Scheduled
데이터가 계속 옴 → Streaming
```

---

### 3.3 processSteps (복수 선택, 1~10개)

> "Agent가 수행하는 단계"를 선택합니다. 사람이 하는 단계는 제외하세요.

| 선택지 | 키워드 | 선택 기준 |
|--------|--------|----------|
| **검색, API 조회, 문서 읽기 (정보 수집)** | 읽기, 조회, 검색, 수집 | 외부에서 데이터를 가져와야 하는가? |
| **요약, 분류, 패턴 인식 (분석/이해)** | 요약, 분류, 분석, 파악 | 데이터를 이해하고 구분해야 하는가? |
| **의사결정, 평가, 우선순위 (추론/판단)** | 결정, 승인, 평가, 순위 | Agent가 직접 판단을 내려야 하는가? |
| **문서, 코드, 응답 작성 (생성/작성)** | 작성, 생성, 만들기 | 새로운 콘텐츠를 만들어야 하는가? |
| **품질 확인, 자기 수정 (검증/반복)** | 검증, 확인, 수정, 재시도 | 결과물을 스스로 검토해야 하는가? |

**핵심 원칙: "Agent가 하는 일만 고르세요"**

```
Agent가 하는 일 → processSteps에 포함
사람이 하는 일  → humanLoop에서 다룸
```

**선택 개수 가이드:**
- 1~2개: 단순 자동화 (단일 Agent)
- 3~4개: 일반적인 Agent (워크플로우 패턴 가능)
- 5개 전부: 복잡한 자율형 Agent (멀티 에이전트 패턴 가능)

---

### 3.4 outputTypes (복수 선택, 1~10개)

> "Agent가 만들어내는 최종 결과물"을 선택합니다.

| 선택지 | 판단 기준 | 예시 |
|--------|----------|------|
| **답변, 요약, 설명 (텍스트 응답)** | 실시간 대화형 출력인가? | 챗봇 답변, 즉석 요약 |
| **이메일, 리포트, 코드 (문서/보고서)** | 저장/전달되는 문서인가? | 이메일 초안, 주간 보고서, 코드 파일 |
| **JSON, 분류 결과, 점수 (구조화된 데이터)** | 정형화된 데이터인가? | 분류 라벨, 점수, 메타데이터 |
| **DB 업데이트, 티켓 생성, 알림 (외부 시스템 변경)** | 외부 시스템을 변경하는가? | Jira 생성, DB 업데이트, Slack 알림 |

**구분이 헷갈릴 때:**

```
같은 텍스트 출력이라도...
  실시간 대화    → "텍스트 응답"
  저장/전달 문서  → "문서/보고서"

같은 데이터라도...
  사람이 읽는 용도  → "텍스트 응답" 또는 "문서/보고서"
  시스템이 쓰는 용도 → "구조화된 데이터"
```

---

### 3.5 humanLoop (단일 선택)

> "Agent의 실행 흐름에 사람이 개입하는가?"를 기준으로 선택합니다.

| 선택지 | 판단 기준 | 적합한 상황 |
|--------|----------|-----------|
| **완전 자동 (None)** | 결과가 본인에게만 가거나, 틀려도 치명적이지 않음 | 내부 참고용, 개인 도구, 로그 처리 |
| **실행 전 승인 필요 (Review)** | 결과가 외부(고객, 상사)에게 전달됨 | 이메일 발송, 보고서 제출, 결제 처리 |
| **불확실할 때만 개입 (Exception)** | 대부분 자동이지만 예외 상황에서 사람 필요 | 대량 처리 중 이상 케이스만 사람에게 |
| **AI와 함께 작업 (Collaborate)** | 사람과 AI가 함께 반복적으로 개선 | 문서 공동 작성, 디자인 피드백 루프 |

**핵심 질문: "이 Agent가 실수하면 누가 피해를 보는가?"**

```
본인만 피해     → None
고객/외부 피해   → Review
대부분 괜찮은데 가끔 위험 → Exception
함께 만들어가는 작업      → Collaborate
```

---

### 3.6 errorTolerance (단일 선택)

> "Agent의 결과물에 요구하는 품질 수준"을 선택합니다.
> (humanLoop과 혼동 주의: humanLoop은 "흐름", errorTolerance는 "품질")

| 선택지 | 판단 기준 | 대표 시나리오 |
|--------|----------|-------------|
| **틀려도 괜찮음 (낮은 리스크)** | 참고용이라 틀려도 큰 문제 없음 | 브레인스토밍, 아이디어 초안, 내부 메모 |
| **사람이 검토 후 실행** | 사람이 결과를 보고 판단 후 사용 | 보고서 초안, 코드 제안, 답변 참고 |
| **높은 정확도 필요 (90%+)** | 오류가 직접적인 업무 차질을 유발 | 고객 분류, 일정 조율, 데이터 추출 |
| **매우 높은 정확도 필요 (99%+)** | 오류가 법적/재무적 문제를 유발 | 재무 처리, 법률 검토, 의료 정보 |

**핵심 질문: "틀렸을 때의 결과가 얼마나 심각한가?"**

```
"참고만 할 거야"      → 틀려도 괜찮음
"보고 내가 판단할 거야" → 사람이 검토 후 실행
"바로 적용될 거야"     → 높은 정확도 (90%+)
"돈이나 법이 걸려있어"  → 매우 높은 정확도 (99%+)
```

---

### 3.7 humanLoop vs errorTolerance 구분

이 두 필드가 가장 많이 혼동됩니다. 아래 표로 차이를 명확히 이해하세요.

| 구분 | humanLoop | errorTolerance |
|------|-----------|---------------|
| **질문** | 실행 흐름에 사람이 끼는가? | 결과물의 품질 기대치는? |
| **대상** | 프로세스 (Process) | 산출물 (Output) |
| **예시** | "발송 전에 확인할게" (Review) | "90% 이상 맞아야 해" (높은 정확도) |

**동일한 조합이 아닌 경우가 자연스럽습니다:**

| 사례 | humanLoop | errorTolerance |
|------|-----------|---------------|
| 내부 Q&A 봇 | None (자동) | 사람이 검토 후 실행 (사용자가 판단) |
| 고객 이메일 분류 | Review (승인 필요) | 높은 정확도 (분류가 정확해야 함) |
| 아이디어 브레인스토밍 | Collaborate (함께) | 틀려도 괜찮음 (발산 단계) |

## 4. Guided 실습: 주제 1 — 고객 문의 자동 분류 및 응답 초안 생성

> 가장 범용적인 주제. 워크샵 첫 실습으로 추천합니다.

### Step 1 입력값 및 선택 이유

#### painPoint
> 하루 평균 200건의 고객 문의 이메일이 들어오는데, 담당자가 수동으로 유형(환불/배송/제품불량/일반문의)을 분류하고 초안을 작성하느라 하루 3시간 이상 소요. 긴급 문의 대응이 늦어져 고객 불만 증가

#### inputType → `Event-Driven (이벤트 발생 시)`

| 후보 | 판단 |
|------|------|
| On-Demand | 담당자가 매번 수동 요청? → 자동화 효과 반감 |
| Scheduled | 모아서 일괄 처리? → 긴급 문의 대응 늦어짐 |
| **Event-Driven** ✅ | **이메일이 도착하는 순간이 트리거 → 긴급 문의 즉시 감지 가능** |

> **왜?** 트리거를 생각하세요. 고객 문의가 "들어올 때" 바로 처리해야 하니까 Event-Driven입니다. 만약 "매일 아침 어제 문의를 일괄 처리"라면 Scheduled가 맞습니다.

#### processSteps → 3개 선택

| 순서 | 선택 | 이유 |
|------|------|------|
| 1️⃣ | **정보 수집** | 이메일 내용을 읽고, 고객 주문 이력을 DB에서 조회 |
| 2️⃣ | **분석/이해** | 문의 유형(환불/배송/불량/일반)을 분류하고 긴급도 판단 |
| 3️⃣ | **생성/작성** | 분류 결과를 바탕으로 응답 초안 생성 |

| 선택하지 않은 것 | 이유 |
|-----------------|------|
| 추론/판단 | 환불 승인 같은 최종 판단은 사람이 함 |
| 검증/반복 | 응답 초안 검증은 담당자(사람)가 함 |

> **원칙:** "Agent가 하는 일"만 고르세요. "담당자 검토 후 발송"은 사람이 하는 일이니 processSteps에 넣지 않습니다.

#### outputTypes → 2개 선택

| 선택 | 이유 |
|------|------|
| **구조화된 데이터** | 분류 라벨(`환불/배송/불량/일반`)과 긴급도 태그 |
| **문서/보고서** | 응답 초안은 이메일 형태의 문서 |

| 선택하지 않은 것 | 이유 |
|-----------------|------|
| 텍스트 응답 | 챗봇 실시간 답변이 아니라 이메일 "문서"를 작성하는 것 |
| 외부 시스템 변경 | CRM 자동 저장까지 포함한다면 선택하지만, 이번 범위는 초안 생성까지 |

> **구분 팁:** 실시간 대화형이면 "텍스트 응답", 저장/전달되는 결과물이면 "문서/보고서"

#### humanLoop → `실행 전 승인 필요 (Review)`

| 후보 | 판단 |
|------|------|
| None | 고객 대면 이메일을 검토 없이 발송? → 리스크 과다 |
| Exception | AI가 확신 있다고 보낸 메일이 틀리면? → 고객 신뢰 하락 |
| Collaborate | 200건을 매번 함께 작업하면 시간 절약 효과 미미 |
| **Review** ✅ | **AI가 초안 생성, 담당자가 빠르게 확인 후 발송 → 속도와 안전성 균형** |

> **판단 기준:** "실수했을 때 감당할 수 있는가?" 고객에게 직접 가는 결과물은 Review가 안전합니다.

#### errorTolerance → `높은 정확도 필요 (90%+)`

| 후보 | 판단 |
|------|------|
| 틀려도 괜찮음 | 분류가 자주 틀리면 수동 작업과 차이 없음 |
| 사람이 검토 후 실행 | Review도 하지만, 분류 자체의 기대 정확도를 말하는 항목 |
| **높은 정확도 (90%+)** ✅ | **분류 오류 10% 이내면 담당자가 빠르게 수정 가능. 긴급 누락은 불허** |
| 매우 높은 정확도 (99%+) | 이메일 분류는 재무/법률 수준의 정밀도까지 불필요 |

> **판단 기준:** "틀렸을 때 결과가 얼마나 심각한가?" 돈이 오가면 99%+, 고객 대면은 90%+, 내부 참고용은 낮은 리스크.

## 5. Guided 실습: 주제 2 — 내부 기술 문서 기반 Q&A 어시스턴트

> RAG + Agent 패턴을 다루는 기술팀 대상 주제.

### Step 1 입력값 및 선택 이유

#### painPoint
> 사내 기술 문서가 Confluence, GitHub Wiki, Notion에 분산되어 있어 신규 입사자나 타팀 개발자가 필요한 정보를 찾는 데 평균 30분 이상 소요. 문서 최신화 여부도 확인 어려움

#### inputType → `On-Demand (사용자 요청)`

| 후보 | 판단 |
|------|------|
| Event-Driven | Q&A는 이벤트가 트리거하는 게 아님 |
| Scheduled | 정해진 시간에 질문을 모아서 답변? → 말이 안 됨 |
| **On-Demand** ✅ | **개발자가 궁금할 때 질문 → 즉시 답변. 전형적인 On-Demand** |

> **왜?** "누가 시작하는가"를 보세요. 사람이 필요할 때 요청하면 On-Demand, 시스템이 감지하면 Event-Driven입니다.

#### processSteps → 3개 선택

| 순서 | 선택 | 이유 |
|------|------|------|
| 1️⃣ | **정보 수집** | 3개 플랫폼에서 관련 문서를 검색하고 내용을 읽음 (RAG의 Retrieval) |
| 2️⃣ | **분석/이해** | 검색된 문서 중 질문과 관련된 핵심 내용 추출 |
| 3️⃣ | **생성/작성** | 추출된 내용으로 답변 + 출처 링크 생성 (RAG의 Generation) |

| 선택하지 않은 것 | 이유 |
|-----------------|------|
| 추론/판단 | Q&A 봇은 정보 전달이 목적, 판단/결정이 아님 |
| 검증/반복 | 출처 확인은 생성 단계에서 처리 가능. 별도 검증 루프 불필요 |

> **비교:** 주제 1과 processSteps 선택이 동일하지만 이유가 다릅니다. 같은 선택지도 맥락에 따라 의미가 달라집니다.

#### outputTypes → 2개 선택

| 선택 | 이유 |
|------|------|
| **텍스트 응답** | 질문에 대한 즉석 답변 → 대화형 텍스트가 메인 |
| **구조화된 데이터** | 출처 문서 링크 목록, 신뢰도 점수 |

| 선택하지 않은 것 | 이유 |
|-----------------|------|
| 문서/보고서 | 저장되는 문서가 아니라 실시간 대화형 답변 |
| 외부 시스템 변경 | Q&A 봇은 읽기 전용, 외부 시스템을 변경하지 않음 |

> **주제 1과 비교:** 주제 1은 "이메일=문서"를, 여기서는 "즉석 답변=텍스트 응답"을 골랐습니다. 같은 텍스트 출력이라도 저장/전달용이면 문서, 실시간 대화면 텍스트 응답.

#### humanLoop → `완전 자동 (None)`

| 후보 | 판단 |
|------|------|
| **None** ✅ | **질문한 직원 본인이 답변을 받아서 스스로 판단. 별도 승인자 불필요** |
| Review | 매 질문마다 승인? → Q&A 봇의 존재 의미 없어짐 |
| Exception | 가능하지만 내부 문서 Q&A는 틀려도 치명적이지 않아 과잉 |

> **왜?** "결과물이 누구에게 가는가"를 보세요. 본인이 직접 쓰면 None, 고객이나 외부에 가면 Review.

#### errorTolerance → `사람이 검토 후 실행`

| 후보 | 판단 |
|------|------|
| 틀려도 괜찮음 | 답변이 자주 틀리면 신뢰를 잃고 아무도 안 씀 |
| **사람이 검토 후 실행** ✅ | **답변을 보고 개발자가 직접 판단 후 적용. 출처 링크로 검증 가능** |
| 높은 정확도 (90%+) | 내부 참고용이라 구현 난이도가 과도하게 높아짐 |

> **humanLoop과의 관계:** 흐름은 자동(None)이지만, 사용자가 답변을 검토하고 쓸지 말지 결정합니다. 이 조합이 자연스럽습니다.

### Step 1 입력값 및 선택 이유

#### painPoint
> 매주 금요일 팀원 5명이 각자 Jira 티켓, Slack 채널, Git 커밋 내역을 수동으로 취합하여 주간 보고서를 작성하는 데 1인당 1시간 소요. 포맷도 매번 달라서 팀장 취합에 추가 30분 소요

#### inputType → `Scheduled (정해진 시간)`

| 후보 | 판단 |
|------|------|
| Event-Driven | 보고서는 이벤트가 아니라 주기적 작업 |
| On-Demand | 가능하지만 매주 금요일 자동 실행이 더 효율적 |
| **Scheduled** ✅ | **"매주 금요일"이라는 명확한 주기 → 전형적인 Scheduled** |

> **왜?** pain point에 "매주", "매일", "매월" 같은 키워드가 있으면 Scheduled입니다. "매번 요청이 올 때마다"는 On-Demand.

#### processSteps → 4개 선택

| 순서 | 선택 | 이유 |
|------|------|------|
| 1️⃣ | **정보 수집** | Jira API, Slack API, GitHub API에서 금주 데이터 수집 |
| 2️⃣ | **분석/이해** | 티켓/메시지/커밋을 팀원별/프로젝트별로 분류하고 핵심 요약 |
| 3️⃣ | **생성/작성** | 요약 결과를 통합 주간보고서 포맷으로 작성 |
| 4️⃣ | **검증/반복** | 티켓 누락 여부 교차 확인, 요약 내용 재검토 |

| 선택하지 않은 것 | 이유 |
|-----------------|------|
| 추론/판단 | 보고서는 사실 정리이지 판단/결정이 아님 |

> **주제 1, 2와 비교:** 이 주제만 검증/반복이 추가되었습니다. "티켓 누락 불허"라는 요구사항 때문입니다. pain point의 요구사항이 processSteps 선택에 영향을 줍니다.

#### outputTypes → 2개 선택

| 선택 | 이유 |
|------|------|
| **문서/보고서** | 주간보고서 자체가 리포트 형태 |
| **구조화된 데이터** | 팀원별 업무 요약은 구조화해야 통합 가능 |

| 선택하지 않은 것 | 이유 |
|-----------------|------|
| 텍스트 응답 | 대화형이 아니라 문서 산출물 |
| 외부 시스템 변경 | 보고서 생성이 목적 (Slack 전송까지 한다면 추가) |

> **범위에 따라 달라지는 예:** "보고서 완성 후 Slack에 자동 전송"까지 포함하면 "외부 시스템 변경"도 선택해야 합니다.

#### humanLoop → `실행 전 승인 필요 (Review)`

| 후보 | 판단 |
|------|------|
| None | 팀장 이름으로 나가는 보고서를 검토 없이? → 리스크 |
| **Review** ✅ | **자동 생성 후 팀장이 훑어보고 발송 → 30분 취합이 5분 검토로 단축** |
| Exception | 보고서는 매번 사람이 한 번은 봐야 자연스러움 |
| Collaborate | 매주 반복 작업에 협업 모드는 과잉 |

> **패턴 발견:** 주제 1(이메일)과 주제 3(보고서) 모두 Review. "외부로 나가는 결과물은 Review"라는 패턴이 보이시나요?

#### errorTolerance → `사람이 검토 후 실행`

| 후보 | 판단 |
|------|------|
| 틀려도 괜찮음 | 티켓 누락은 안 되므로 느슨함 |
| **사람이 검토 후 실행** ✅ | **팀장이 최종 검토. 요약 표현의 완벽함보다 팩트(티켓 목록)의 정확성이 핵심** |
| 높은 정확도 (90%+) | 내부 보고서에 90%+ 강제하면 구현 복잡도만 상승 |

### 주제 선정 체크리스트

자유 주제를 정할 때 아래 질문에 답할 수 있어야 합니다.

- [ ] **정량화**: 현재 몇 시간/몇 명이 소요되는가?

- [ ] **명확성**: "잘 처리된 결과"를 어떻게 판단하는가?

### Mock 데이터 설계 원칙

| 원칙 | 설명 |
|------|------|
| **10건 내외** | 너무 적으면 패턴이 안 보이고, 너무 많으면 시간 낭비 |
| **Happy Path 60%** | 정상 케이스가 대부분이어야 기본 동작 확인 가능 |
| **Edge Case 20%** | 분류가 애매하거나 복합적인 케이스 |
| **긴급/예외 20%** | 긴급 감지, 에러 처리 테스트용 |

### 선택이 막힐 때 참고하는 의사결정 트리

**inputType**
```
업무가 언제 시작되나요?
├── 무언가 발생했을 때  → Event-Driven
├── 정해진 주기마다    → Scheduled
├── 내가 원할 때       → On-Demand
├── 데이터가 계속 옴   → Streaming
└── 특정 조건 충족 시  → Conditional
```

**humanLoop**
```
Agent가 실수하면?
├── 나만 불편          → None
├── 고객/상사가 피해    → Review
├── 대부분 괜찮은데 가끔 위험 → Exception
└── 함께 만들어가는 작업     → Collaborate
```

**errorTolerance**
```
틀렸을 때 결과가?
├── 별로 안 심각       → 틀려도 괜찮음
├── 내가 보고 걸러냄   → 사람이 검토 후 실행
├── 업무에 직접 차질   → 높은 정확도 (90%+)
└── 돈/법/안전 관련    → 매우 높은 정확도 (99%+)
```
